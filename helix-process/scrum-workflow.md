# Scrum HELIX ワークフロー

## 概要

Scrum は、ユーザーと要件をすり合わせながら反復的に開発するモード（アジャイル Scrum）。スプリント単位で機能（インクリメント）を作り、ユーザーのフィードバックで要件を継続的に調整する。作るものは概ね決まっているが、要件をユーザーと合わせながら固めて進めたい場合に使う。

## 業界標準との関係

> 注: スクラムガイドに準拠する。ロール（プロダクトオーナー / スクラムマスター / 開発者）、イベント（スプリント / スプリントプランニング / デイリースクラム / スプリントレビュー / スプリントレトロスペクティブ）、作成物（プロダクトバックログ / スプリントバックログ / インクリメント）。

## 入口判定

| 状況 | Scrum を使う理由 |
|---|---|
| 作るものは概ね明確だが要件をユーザーと合わせたい | 反復しながら要件を固める |
| ユーザーフィードバックで継続調整したい | スプリントごとにレビューで反映する |
| 早く動くものを見せて方向を確認したい | インクリメントを反復的に届ける |

要件・設計・契約が最初から確定しているなら Forward HELIX、成立するか未確定なら Discovery、既存資産の逆引きなら Reverse。

## 基本フロー（スプリント反復）

```
プロダクトバックログ
   → スプリントプランニング
   → 開発（デイリースクラムで日次同期）
   → スプリントレビュー（ユーザーと要件すり合わせ）
   → スプリントレトロスペクティブ
   → インクリメント（機能完成）
   ↺ 次スプリントへ反復
```

## ロール・イベント・作成物

| 区分 | 要素 |
|---|---|
| ロール | プロダクトオーナー / スクラムマスター / 開発者 |
| イベント | スプリント / プランニング / デイリースクラム / レビュー / レトロスペクティブ |
| 作成物 | プロダクトバックログ / スプリントバックログ / インクリメント |

## Vモデル体系への昇華フロー（重要）

スクラムで作った機能（インクリメント）は、最終的に Forward（Vモデル L0–L14）のドキュメント体系に昇華する。スクラムの速さ（反復・フィードバック）と Vモデルの厳格さ（ドキュメント体系・トレーサビリティ）を両立させるための接続フロー。

```
スクラム反復開発 → 機能（インクリメント）完成
   ↓ ドキュメント化（Reverse fullback を活用）
完成機能から成果物を起こす（F0–F4: evidence / contracts / as-is review / handover checklist / routing）
   ↓ Vモデル体系へ統合
L1 要求定義 / L3 要件定義 / L4–L6 設計 / L8–L9 テスト の各工程ドキュメントに整える
```

### 昇華の手順

1. スプリントで機能（インクリメント）が完成する。
2. `helix reverse fullback`（完了実装から文書整合と handover checklist を作る）でドキュメント化する。
3. 起こしたドキュメントを Forward の L0–L14 各工程ファイルの枠組み（`helix-process/L*.md`）に整理する。
4. 以後その機能も Vモデルのドキュメント体系（双方向 trace・ゲート）で管理する。

### 昇華先の対応

| 完成機能から起こす内容 | Vモデル昇華先 |
|---|---|
| 確定した要求・要件 | L1 要求定義 / L3 要件定義 |
| 実装された設計（方式・機能・データ） | L4 基本設計 / L5 詳細設計 / L6 機能設計 |
| 実装済みのテスト | L8 結合テスト / L9 総合テスト |
| 運用・受入・文書整合 | L11–L14 |

これにより、スクラムで先に動くものを作っても、最終的に Vモデルの正式なドキュメント体系に乗り、トレーサビリティが確保される。

## 起票する PLAN kind

完成機能の文書化は Reverse の `fullback` kind（F0–F4）を起票し、L0–L14 体系へ昇華する。逸脱と kind の対応は deviation-plan-map.md を参照。
