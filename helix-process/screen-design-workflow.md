# 画面設計ワークフロー（UI / ワイヤーフレーム）

## 概要

L2 画面設計・フロントUI 工程の専門ワークフロー。UI のワイヤーフレームを中心に画面構造を設計する。HELIX の FE 弱点（FE 専用 detector が定義先行で未実装）を補うため、画面設計を専門化する。

## 位置づけ

- Vモデル L2 工程（画面設計・フロントUI / ワイヤーモック作成）の専門化。
- 入口判定モードではなく、L2 工程内の進め方を専門化したもの。
- 成果物は `L2-ui-design.md` の PLAN（画面一覧 / 画面遷移 / ワイヤーフレーム / UI要素）に対応する。
- 右腕の L10 フロントデザイン（UX ビジュアル磨き上げ）とペアになる。

## 業界標準プロセス

```
情報設計（IA）→ 画面一覧・遷移フロー → ワイヤーフレーム（Low-Fi → High-Fi、モノクロ中心）
   → モックアップ → プロトタイプ → ユーザビリティテスト → コンポーネント化
```

原則: 1画面1目的 / 1画面1機能、情報の優先順位の整理を最重要とする。

## フロー

1. **情報設計（IA）**: 情報の整理・構造化
2. **画面一覧・遷移フロー**: 画面の洗い出しと遷移定義
3. **ワイヤーフレーム Low-Fi**: 骨格（色・装飾なし、レイアウトと優先順位に集中）
4. **ワイヤーフレーム High-Fi**: 詳細化
5. **モックアップ**: 見た目の確認
6. **プロトタイプ**: 操作・遷移の体験
7. **ユーザビリティテスト**: 5〜7人で操作を観察
8. **コンポーネント化**: 再利用単位へ

## HELIX 検証観点（FE detector）

| detector | 検証 | 現状 |
|---|---|---|
| `state-transition-drift` | 画面遷移の設計と実装の乖離 | 定義先行・未実装（本モードで実装目標を明確化） |
| `mock-promotion` | モック → component_impl の昇格が append-only か | 同上 |

## Forward 接続

確定したワイヤーフレーム・画面遷移は L2 の成果物として L3 要件定義・L4 基本設計（画面設計確定）へ渡す。モックは G2 で凍結し、L7 実装で component_impl へ append-only に昇格する。
