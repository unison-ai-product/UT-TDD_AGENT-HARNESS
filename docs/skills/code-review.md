# コードレビュースキル

## 適用タイミング

このスキルは以下の場合に読み込む：
- コードレビュー実施時
- PRテンプレート作成時
- レビュー文化構築時

---

## 1. レビューの目的

```
1. バグの早期発見
2. コード品質の維持
3. 知識の共有
4. 設計の改善
5. 一貫性の確保
```

### レビューで見つけるべきもの

| 優先度 | 項目 |
|--------|------|
| 高 | セキュリティ問題、バグ、データ損失リスク |
| 中 | 設計問題、パフォーマンス、エラーハンドリング |
| 低 | コードスタイル、命名、コメント |

---

## 2. レビュー観点

### 機能

```
[ ] 仕様を満たしているか
[ ] エッジケースは考慮されているか
[ ] エラーハンドリングは適切か
[ ] 既存機能への影響はないか
```

### 設計

```
[ ] 責務は適切に分離されているか
[ ] 依存関係は適切か
[ ] 拡張しやすい設計か
[ ] 既存パターンと整合しているか
[ ] 過剰な設計ではないか
```

### セキュリティ

```
[ ] SQLインジェクション対策
[ ] XSS対策
[ ] 認証・認可は適切か
[ ] 機密情報の扱い
[ ] 入力バリデーション
[ ] ログに機密情報が含まれていないか
```

### パフォーマンス

```
[ ] N+1問題はないか
[ ] 不要なクエリはないか
[ ] メモリリークの可能性
[ ] 無限ループの可能性
[ ] 大量データ処理の考慮
```

### テスト

```
[ ] テストは十分か
[ ] 正常系・異常系・境界値
[ ] テストは読みやすいか
[ ] モックは適切か
```

### コード品質

```
[ ] 読みやすいか
[ ] 命名は適切か
[ ] 重複はないか
[ ] 複雑すぎないか
[ ] コメントは必要十分か
```

## 3. レビューコメントの書き方

### コメント種別

```markdown
# 必須（ブロッキング）
[Must] SQLインジェクションの可能性があります。
パラメータ化クエリを使用してください。

# 推奨（非ブロッキング）
[Should] この関数は長いので分割を検討してください。

# 提案（任意）
[Nit] 変数名は `userData` より `userProfile` の方が明確かも。

# 質問
[Question] この処理の意図を教えてください。

# 称賛
[Good] このエラーハンドリングは素晴らしいですね！
```

### 良いコメントの例

```markdown
# 悪い例
「これはダメです」
「なぜこう書いたの？」

# 良い例
「この処理だと、userがnullの場合に例外が発生します。
事前にnullチェックを入れるのはいかがでしょうか？」

「パフォーマンスの観点から、このループ内でのDB呼び出しは
ループ外でまとめて取得する方が良いかもしれません。」
```

### フィードバックの原則

```
1. コードを批判し、人を批判しない
   - 悪: 「あなたのコードは...」
   - 良: 「このコードは...」

2. 理由を説明する
   - 悪: 「これは良くない」
   - 良: 「これは〇〇の理由で問題になる可能性があります」

3. 改善案を提示する
   - 悪: 「直してください」
   - 良: 「〇〇のようにすると良いかもしれません」

4. 質問形式を使う
   - 悪: 「〇〇すべき」
   - 良: 「〇〇するのはいかがでしょうか？」

5. 良い点も伝える
   - 「このエラーハンドリングは丁寧で良いですね」
```

## 4. レビュイーの心得

### PR作成時

```
1. 小さく保つ
   - 300行以下を目標
   - 1PR = 1目的

2. 説明を書く
   - 何を変更したか
   - なぜ変更したか
   - どうテストしたか

3. セルフレビュー
   - 提出前に自分で確認
   - diffを見直す

4. テストを通す
   - CIが通る状態で提出
   - 手動テストも実施
```

### PRテンプレート

```markdown
## 概要
<!-- 変更の概要を簡潔に -->

## 変更理由
<!-- なぜこの変更が必要か -->

## 変更内容
- [ ] 機能A追加
- [ ] テスト追加
- [ ] ドキュメント更新

## テスト
- [ ] 単体テスト通過
- [ ] ローカルで動作確認

## チェックリスト
- [ ] セルフレビュー済み
- [ ] テスト追加/更新済み
- [ ] ドキュメント更新済み（必要な場合）

## 関連チケット
Closes #123
```

## 5. レビュアーの心得

### 姿勢

```
1. 建設的に: 改善を目指す、攻撃しない
2. 迅速に: 24時間以内にレビュー
3. 一貫性を持つ: 人によって基準を変えない
4. 教育的に: 知識を共有する、理由を説明する
```

### 承認基準

```
承認（Approve）: 問題なし、または軽微な指摘のみ（Nit）
修正依頼（Request Changes）: 重大な問題あり、修正後に再レビュー必要
コメント（Comment）: 質問がある、議論が必要
```

## 6. UT-TDD ゲート連携

UT-TDD の V-model においてレビューは gate を通じて管理される。

- **G4 実装凍結ゲート**: Critical/High 指摘 0 件達成が必須条件
- **G6 RC 判定ゲート**: OWASP Top10 チェック完了、FIXME/TODO 0 件
- **intra_runtime_subagent レビュー**: `claude-only` / `standalone` mode では cross-agent 代替として `code-reviewer` subagent を通す

### 完了判定基準

```
[ ] Critical/High指摘 0件（全て解消済み）
[ ] セキュリティ: OWASP Top10チェックリスト 未対応 0件
[ ] FIXME/TODO 0件（rg 'FIXME|TODO' 対象ファイル）
[ ] CI通過（lint + test + build）
```

## 7. レビューコスト最適化

変更の重要度に応じてレビュー担当モデルを切り替え、品質を維持しつつコストを最適化する。

### 重要度別のモデル選択

| リスク | 変更例 | 推奨アプローチ |
|--------|--------|----------------|
| 低リスク | typo、コメント修正、import整理 | Haiku または自動lint |
| 中リスク | ロジック変更、テスト追加 | Sonnet レビュー |
| 高リスク | 認証、決済、DBマイグレーション | 上位モデル必須 |

### ルーティング手順

```
1. 変更内容からリスクを評価（低/中/高）
2. リスク帯で一次レビュアーを選択
3. 認証/決済/DB migration が含まれる場合は上位モデルに昇格
4. Critical/High 指摘が残る場合は必ず上位モデルで再レビュー
```
