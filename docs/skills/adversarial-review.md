# AI Adversarial Review スキル

## 適用タイミング

### 自動発火（自発的に起動）

| 条件 | 発火タイミング | 根拠 |
|------|-------------|------|
| セキュリティゲート条件該当 | L2 設計凍結ゲート | セキュリティゲート強制条件（認証・決済・PII・外部API・インフラ構成変更） |
| ADR 作成 | ADR ドラフト完了後 | 後戻りコストが高い技術判断には反証が必要 |

### 任意発火（PM 判断）

| 条件 | 判断基準 |
|------|---------|
| 後戻りコストが高い設計判断 | DB設計の根幹・認証方式・外部API統合方針など |
| フレームワーク・運用ルールの大規模変更 | UT-TDD 自体の構造変更など |

### 責務境界

- **adversarial-review (本スキル)**: 批判的レビュー手法 (G2 固有)
- **security**: 脆弱性・OWASP・秘密情報・AI生成コード品質
- **compliance**: 法令遵守・ライセンス・規制対応
- **threat-model**: STRIDE/DREAD による脅威モデル作成

### スキップ条件

- サイジング S（小規模）のタスク
- 既存ルールの機械的適用
- 同一プロジェクトで同一テーマの adversarial-review が実施済み

### 既存レビュー体制との責務分離

| レビュー手段 | 目的 | コスト | いつ |
|------------|------|--------|------|
| TL壁打ち | 技術的妥当性の確認（1案の検証） | 低 | プランモード Phase 3 |
| ut-tdd review | コード/仕様の品質チェック | 中 | フェーズ遷移時 |
| **adversarial-review** | **「本当にそれでいいのか？」の反証（複数案の対立検証）** | **高** | **高リスク判断時のみ** |

## 1. 前提条件チェック

```
適用必須（上記「自動発火」条件に該当）：
□ セキュリティゲート条件に該当する設計変更
□ ADR を作成した（後戻りコストが高い判断）
□ PM が「後戻りコストが高い」と判断

スキップ可能：
□ サイジング S（小規模タスク）
□ 既存ルールの機械的適用
□ 同一テーマで実施済み
```

---

## 2. 三者分離アーキテクチャ

```
議長（Coordinator）
    │
    ├─→ Proposer A（独立インスタンス）
    │      └─ 独立コンテキスト
    │
    └─→ Proposer B（別モデル）
           └─ 独立コンテキスト
```

### 役割分離ルール

| 役割 | 責務 | 禁止事項 |
|------|------|----------|
| 議長 | 進行・裁定 | 自ら提案を出す |
| Proposer | 独立提案 | 他Proposerの出力を見る |

### 議長が提案者を兼ねる場合

1. 別インスタンスを起動（Task tool / `ut-tdd codex --role tl-advisor`）
2. 両提案者へのプロンプトは**同一文面**
3. 結果の要約は**引用ベース**（言い換え禁止）

---

## 3. コアサイクル

```
┌─────────────────────────────────────────┐
│  1. Propose（各AIが独立して案を作成）    │
│     ↓                                   │
│  2. Collide（差異・前提・評価軸を特定） │
│     ↓                                   │
│  3. Challenge（暫定結論への批判的検証） │
│     ↓                                   │
│  4. Decide（決定＋反対意見を記録）      │
└─────────────────────────────────────────┘
         ↑                    │
         └────────────────────┘
           批判で修正発生時（最大3回）
```

### 各ステップの出力

| ステップ | 出力 | 責任者 |
|----------|------|--------|
| Propose | 提案書（案＋根拠＋前提） | proposers |
| Collide | 差異表（同意点/相違点/理由） | coordinator |
| Challenge | 批判リスト（指摘＋代替案） | adversarial_reviewer |
| Decide | 決定書（結論＋反対意見＋根拠） | owner |

---

## 4. 独立性の確保

### 必須要件

- 評価軸を2つ以上定義してから開始
- proposer同士は結論を共有しない
- 評価軸は議論開始前に固定・公開

### オプション強化

- 要約係を議長から分離
- 二重ブラインド化
- プロンプト凍結（途中変更不可）

### 利益相反の回避

- 自分が書いたコードのレビューは別AIが担当

---

## 5. 裁定基準（優先順位）

```
1. 倫理基準（法令・規約違反は絶対NG）
2. リスク最小化（被害規模を優先）
3. 可逆性（戻せる選択肢を優先）
4. ビジネス価値（同等なら価値が高い方）

競合時は上位基準が優先
```

## 6. 監査ログ

### 必須記録フィールド

```yaml
decision_record:
  decision_id: "ADR-YYYYMMDD-NNN"
  decision_summary: "何を決めたか"
  dissent_summary: "反対意見の要約"
  arbitration_basis: "裁定理由（どの基準を適用）"
  timestamp: "ISO 8601形式"
```

### 保管要件

- 保管場所: `.ut-tdd/audit/decisions/`
- 保管期間: 1年以上
- 形式: 第三者が追跡可能な構造化データ

### 追加ログ（推奨）

- プロンプト全文
- 各提案者の出力全文
- 要約・裁定の全文

## 7. 緊急時プロトコル

### 適用条件

- 障害対応など即時判断が必要
- 判断が24時間以内に可逆

### ガードレール

| 項目 | ルール |
|------|--------|
| 事後レビュー | 24時間以内に実施 |
| 月次監査 | 例外適用を集計 |
| エスカレーション | 月3回超→プロセス改善会議 |

---

## 8. チェックリスト

### 開始前

```
□ 適用条件を確認（必須/スキップ）
□ 評価軸を2つ以上定義
□ 提案者の独立性を確保
□ 議長と提案者を分離
```

### 進行中

```
□ 両提案者に同一プロンプトを送信
□ 結果は引用ベースで要約
□ 差異表を作成
```

### 決定時

```
□ 裁定基準の優先順位を適用
□ 反対意見を記録
□ 決定書を作成
□ .ut-tdd/audit/decisions/ に保管
```

## 9. アンチパターン

| パターン | 問題 | 対策 |
|----------|------|------|
| 議長が提案者を兼ねる | 利益相反 | 別インスタンス起動 |
| 片方に有利なプロンプト | バイアス混入 | 同一文面を使用 |
| 評価軸の事後変更 | 恣意的裁定 | 事前固定・公開 |
| 反対意見の無視 | 盲点の温存 | 必ず記録・検討 |
| ログなし | 追跡不能 | 全文ログ保管 |
