# 設計書スキル

## 適用タイミング

このスキルは以下の場合に読み込む:
- 新規プロジェクト開始時
- 企画書から設計書を作成する時
- 設計レビュー時

---

## 1. 設計フロー

```
企画書（Why/What）
    ↓
★事前調査（強制条件に該当する場合は必須）
    ↓  調査レポート → 設計の入力に追加
基本設計（What/How概要）
    ↓
詳細設計（How詳細）
    ↓
実装
```

### V-model 4 artifact 対応

```
① 設計  ←→  ③ テスト設計
② 実装コード  ←→  ④ テストコード
① 設計  →  ② 実装コード
③ テスト設計  →  ④ テストコード
```

設計フェーズで作成する artifact は **① 設計のみ** とし、対応する **③ テスト設計 artifact は別文書として並行作成** する。

- L2 全体設計: 対応する **総合テスト設計（D-TEST-DESIGN-SYS、③ テスト設計 artifact、別文書）** を別途作成
- L3 詳細設計: 対応する **結合テスト設計（D-TEST-DESIGN-INT）** と **単体テスト設計（D-TEST-DESIGN-UNIT）** を別途作成
- 設計書の frontmatter または冒頭には `test_design_doc: docs/test-design/PLAN-XXX-*-design.md` を明記し、③ テスト設計への reference を固定する
- 4 artifact を別文書・双方向 trace で扱う前提

### L2 開始条件

設計を開始する前に以下を確認する:
- 事前調査強制条件に該当する場合、**事前調査が完了していること**（調査メモが `docs/research/` に存在する）
- 非該当の場合は「事前調査: スキップ（理由: ...）」が明示されていること

### 各フェーズの成果物

| フェーズ | 成果物 | 粒度 |
|---------|--------|------|
| 企画 | 企画書、要件定義 | なぜ作るか、何を作るか |
| 基本設計 | システム構成図、画面一覧、API一覧、ER図 | 全体像、主要機能 |
| 詳細設計 | API仕様、テーブル定義、画面仕様、処理フロー | 実装可能な粒度 |

### L2 成果物（ID管理）

| 成果物ID | 出力先 | 担当スキル |
|---------|--------|-----------|
| D-API | `docs/design/features/{id}/D-API/` | api-contract |
| D-DB | `docs/design/features/{id}/D-DB/` | db |
| D-IA | `docs/design/fe/` | UI 情報設計 |
| D-UX-FLOW | `docs/design/fe/` | UI ユーザーフロー |

## 3. 基本設計から詳細設計

### 詳細設計の粒度

```
基本設計: 「ユーザー登録機能を作る」
詳細設計: 「POST /api/users にname, email, passwordを送ると
          バリデーション後、bcryptでハッシュ化してDBに保存、
          JWTを返す」
```

### 詳細設計書テンプレート（抜粋）

```markdown
# 詳細設計書

## 1. API仕様

### API-001: ユーザー登録
#### エンドポイント
POST /api/v1/users

#### リクエスト
{
  "name": "string (1-100文字)",
  "email": "string (メール形式)",
  "password": "string (8文字以上)"
}

#### レスポンス（成功）
{
  "success": true,
  "data": {
    "id": "string",
    "name": "string",
    "email": "string",
    "createdAt": "datetime"
  }
}

#### レスポンス（エラー）
| コード | 条件 | メッセージ |
|--------|------|-----------|
| 400 | バリデーション | 各フィールドのエラー |
| 409 | メール重複 | Email already exists |

#### 処理フロー
1. リクエストバリデーション
2. メール重複チェック
3. パスワードハッシュ化（bcrypt cost=12）
4. DBにINSERT
5. レスポンス生成
```

## 4. 設計レビュー観点

### 基本設計レビュー

```
□ 要件との整合性
  - 企画書の要件をすべてカバーしているか
  - スコープ外が明確か

□ 技術選定の妥当性
  - 選定理由が明確か
  - チームのスキルとマッチするか
  - 将来の拡張性

□ 実現可能性
  - 期間内に実装可能か
  - 技術的に実現可能か
  - リスクは洗い出されているか
```

### 詳細設計レビュー

```
□ 実装可能な粒度か
  - エンジニアが迷わず実装できるか
  - 曖昧な部分はないか

□ 整合性
  - API仕様とテーブル定義は整合しているか
  - 画面仕様とAPI仕様は整合しているか

□ エラーケース
  - 異常系が網羅されているか
  - エラーメッセージは適切か

□ 非機能要件
  - 性能要件を満たせる設計か
  - セキュリティ考慮されているか

□ テスト設計
  - テスト設計 (D-TEST-DESIGN-INT / D-TEST-DESIGN-UNIT) が別文書として存在する
  - 設計書からテスト設計への双方向 reference が記載されている
```

## 5. 設計Tips

### 良い設計書の特徴

```
読み手が実装できる粒度
図と文章のバランス
変更履歴が追える
用語が統一されている
曖昧な表現がない
```

### 避けるべきこと

```
実装コードをそのまま書く
詳細すぎて変更に弱い
抽象的すぎて実装できない
誰も読まない長大なドキュメント
```

### 設計変更の管理

```markdown
## 変更履歴
| 日付 | 版 | 変更内容 | 変更者 |
|------|-----|---------|--------|
| 2025-01-01 | 1.0 | 初版作成 | xxx |
```

## 6. 基本設計レビューチェックリスト

### アーキテクチャチェック

- [ ] システム構成図が最新
- [ ] 技術スタックの選定理由が記載
- [ ] コンポーネント間の依存関係が明確
- [ ] 単一障害点（SPOF）が特定・対策されている
- [ ] スケーリング戦略が定義されている

### データモデルチェック

- [ ] ER 図が主要エンティティを網羅
- [ ] 正規化レベルが適切（理由付き）
- [ ] インデックス戦略が考慮されている
- [ ] データ量の見積もりがある

### セキュリティチェック

- [ ] STRIDE 脅威分析が完了
- [ ] 認証・認可フローが定義
- [ ] PII の取り扱い方針が明確
- [ ] 暗号化戦略が定義
- [ ] 外部依存のセキュリティ評価

### 整合性チェック

- [ ] 全機能要件が設計に反映されている（トレーサビリティ）
- [ ] API 一覧と画面一覧が機能一覧と対応
- [ ] 非機能要件が設計に反映されている
- [ ] ADR に重要な設計判断が記録されている

## 7. ADR テンプレート

```markdown
# ADR-XXX: {決定タイトル}

## 状況
- 背景
- 課題

## 決定
- 採用する案

## 理由
- 採用理由

## 代替案
- 採用しなかった案

## 影響
- 期待される効果
- 想定される制約

## リスク管理
- リスク識別（技術 / 業務 / 運用 / 法的）
- リスク評価（発生確率 × 影響度 = level）
- リスク対応（回避 / 軽減 / 受容 / 移転）
- 見直し条件
```
