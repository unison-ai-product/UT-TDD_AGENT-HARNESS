# 要件未定義・引継ぎスキル

要件曖昧時や引継ぎ時の確認質問・仮定管理・引継ぎチェックリストを提供。

## 適用タイミング

- 要件が曖昧・未定義の状態でタスクを受けた時
- プロジェクト・タスクの引継ぎ時
- ステークホルダーが変更された時
- 要件変更が発生した時

---

## 1. 要件未定義プロトコル

### 検出条件

以下のいずれかに該当:

- 「適切に」「いい感じに」等の曖昧表現
- 成功条件が不明確
- 対象範囲が不明確
- 優先順位が不明確
- 技術制約が不明確

### 対応フロー

```
要件未定義を検出
    │
    ├─→ 1. 明確化質問リスト作成
    │
    ├─→ 2. 仮定を明示した提案
    │      「〜と仮定して進めます。違う場合はお知らせください」
    │
    └─→ 3. 段階的詳細化
           大まかな方向性 → 詳細化 → 確認
```

### 明確化テンプレート

```markdown
## 要件確認

以下の点を確認させてください：

### 必須確認事項
1. **目的**: この機能で達成したいことは何ですか？
2. **対象**: 誰が使いますか？
3. **成功条件**: どうなれば成功ですか？
4. **スコープ**: 対象外にすることはありますか？

### 任意確認事項
5. **期限**: いつまでに必要ですか？
6. **制約**: 技術的制約はありますか？
7. **優先度**: 他のタスクとの優先度は？

### 仮定（確認待ち）
- [ ] 〜と仮定しています
- [ ] 〜と仮定しています
```

## 2. 仮定ベース進行

### 仮定の明示ルール

- 仮定を明示的に宣言する
- 仮定が覆った場合の影響を評価する
- 仮定の検証タイミングを設定する

### 仮定テンプレート

```yaml
assumptions:
  - id: "A-001"
    content: "認証はJWTで行う"
    impact_if_wrong: "認証基盤の再設計が必要"
    verification_timing: "設計レビュー時"
    status: "unverified"

  - id: "A-002"
    content: "同時接続は100ユーザー以下"
    impact_if_wrong: "スケーリング設計の見直し"
    verification_timing: "要件確認時"
    status: "verified"
```

## 4. 要件変更管理

### 変更インパクト評価

```yaml
change_request:
  id: "CR-001"
  title: "ログイン方法の追加"
  description: "SSO認証を追加"

  impact_analysis:
    scope: "認証モジュール"
    affected_components:
      - auth-service
      - user-service
      - frontend-login
    effort: "2週間"
    risk: "中"
    dependencies:
      - IdP連携設定
      - セキュリティレビュー

  decision:
    status: "approved"
    conditions:
      - "セキュリティレビュー完了後"
    approver: "PM"
    date: "YYYY-MM-DD"
```

### 変更フロー

```
変更要求
    │
    ├─→ インパクト評価
    │     - 影響範囲
    │     - 工数見積
    │     - リスク評価
    │
    ├─→ ステークホルダー承認
    │
    ├─→ 優先度決定
    │     - バックログへ追加
    │     - 緊急対応
    │
    └─→ 実装・検証
```

## 企画書→要件定義 突合チェックリスト

### 突合手順

1. 企画書の全項目を番号付きで列挙
2. 各項目に対応する要件 ID を紐付け
3. 紐付けできない項目 = 漏れ → L1 に追加
4. 要件定義にあって企画書にない項目 = 追加提案 → PO に確認

### 突合マトリクス テンプレート

| # | 企画書の項目 | 要件ID | 受入条件ID | 状態 |
|---|-------------|--------|----------|------|
| 1 | ユーザー登録機能 | REQ-001 | ACC-001 | 対応済み |
| 2 | SNS ログイン | — | — | 漏れ |
| 3 | — | REQ-005 | ACC-005 | 追加提案 |

### G0.5 通過条件（UT-TDD V-model ゲート）

- 漏れ（企画書にあるが要件にない）: 0 件
- 追加提案: PO 承認済み
- 突合マトリクスが `docs/features/{id}/` に保存されている

## 6. 要件定義品質チェックリスト

### 機能要件チェック

- [ ] 全要件に一意の ID（REQ-F-001）が付与されている
- [ ] 各要件に優先度（P0/P1/P2）が設定されている
- [ ] 各要件に受入条件が紐付いている
- [ ] 要件間の依存関係が明示されている
- [ ] 曖昧な表現（適切に、いい感じに、必要に応じて）が排除されている

### 非機能要件チェック

- [ ] 性能要件に数値目標がある（レスポンス ms、同時接続数）
- [ ] セキュリティ要件に認証・認可・PII の方針がある
- [ ] 可用性要件に SLA 目標がある
- [ ] 運用要件にログ・監視・デプロイの方針がある
- [ ] スケーラビリティの考慮がある

### 受入条件チェック

- [ ] 全受入条件が測定可能（数値基準あり）
- [ ] 正常系だけでなく異常系の条件がある
- [ ] 境界値のテスト条件がある
- [ ] パフォーマンス基準が含まれている

### スコープチェック

- [ ] 対象範囲が明確に列挙されている
- [ ] 対象外（スコープ外）が明示されている
- [ ] 将来対応の項目が「Phase 2」等で区分されている

## チェックリスト

### 要件未定義検出時

```
□ 曖昧な表現を特定
□ 明確化質問リストを作成
□ 仮定を明示
□ 確認待ちで進行可能か判断
```

### 引継ぎ時

```
□ 引継ぎチェックリスト完了
□ 開発環境のセットアップ確認
□ 懸案事項の確認
□ 緊急連絡先の確認
```

### 要件変更時

```
□ インパクト評価完了
□ ステークホルダー承認
□ 優先度決定
□ バックログ更新
```
